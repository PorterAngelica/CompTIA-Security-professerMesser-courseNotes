# 🔐 1.4 : Hashing, Salting & Digital Signatures
---
## 1️⃣ What is a Cryptographic Hash?

### 🧾 Definition:

A **hash** converts data into a **fixed-length string** (fingerprint)

👉 Also called:

* Message digest
* Digital fingerprint

---

### 🧠 Analogy:

👉 Think of a hash like a **fingerprint for data**

* Each file = a person
* Hash = their fingerprint
* You can identify the person from the fingerprint but you CANNOT recreate the person from the fingerprint 
* You can identify the file
* ❌ But you **cannot recreate the file** from the hash

---

### ⚠️ IMPORTANT:

* Hashing ≠ Encryption
* ❌ Cannot reverse a hash
* One-way function

---

### 🎯 Purpose:

* Verify **data integrity**
* Detect changes

---

### 🧠 Analogy:

👉 Like checking someone’s fingerprint at a crime scene

* If it matches → it’s the same person
* If it doesn’t → something changed

---

## 2️⃣ Hashing Characteristics

### 🔥 Avalanche Effect:

👉 Small change → completely different hash

Example:

* Change "." → "!"
* Entire hash changes

---

### 🧠 Analogy:

👉 Like changing **one ingredient in a recipe**

* Sugar → Salt → completely different outcome

OR
👉 Like DNA:

* One small change → completely different result

---

### 🔑 Uniqueness:

* Different inputs → different outputs

---

### 🧠 Analogy:

👉 Every person has a **unique fingerprint**

* No two should match

---

### ⚠️ Collision:

Two different inputs → same hash

* Rare (should be)
* Weak algorithms like **MD5** have collisions

---

### 🧠 Analogy:

👉 Two different people → SAME fingerprint

🚨 That’s a huge problem (identity confusion)

---

### 🚫 Avoid:

* MD5 (broken)
* Use stronger algorithms like:

  * SHA-256

---

### 🧠 Analogy:

👉 MD5 = broken fingerprint scanner
👉 SHA-256 = modern accurate scanner

---

## 3️⃣ Real-World Uses of Hashing

### 📥 File Verification

* Download file
* Hash it
* Compare with official hash

👉 If match → file is unchanged

---

### 🧠 Analogy:

👉 Like verifying a package seal

* If seal matches → not tampered
* If different → someone opened it

---

### 🔐 Password Storage

👉 NEVER store passwords:

* In plaintext
* Encrypted

Instead:

* Store **hashes**

---

### 🧠 Analogy:

👉 Website stores your **fingerprint**, not your actual password

---

### 🔄 Login Process:

1. User enters password
2. System hashes it
3. Compare with stored hash
4. Match → access granted

---

### 🧠 Analogy:

👉 “Does this fingerprint match the one we have on file?”

---

## 4️⃣ Salting (VERY IMPORTANT)

### 🧂 Definition:

Add **random data (salt)** to password before hashing

---

### 🧠 Analogy:

👉 Like adding a **random tattoo before taking a fingerprint**

* Same password → different fingerprints
* Makes each user unique

---

### 🎯 Purpose:

* Prevent rainbow table attacks
* Ensure identical passwords ≠ identical hashes

---

### 🧠 Example:

Password = "dragon"

Without salt → same hash for everyone
With salt → different hash per user

---

### 🧠 Analogy:

👉 Without salt:

* Everyone looks identical

👉 With salt:

* Everyone has unique markings

---

## 5️⃣ Rainbow Tables

### 📚 Definition:

Precomputed list of:

* Passwords
* Their hashes

---

### 🧠 Analogy:

👉 Like a **giant cheat sheet or dictionary**

* “Fingerprint X = password 123456”

---

### ⚠️ Problem:

* Can quickly reverse unsalted hashes

---

### 🛡️ Defense:

* Use **salting**

👉 Makes rainbow tables useless

---

### 🧠 Analogy:

👉 Adding salt = scrambling fingerprints
👉 Cheat sheet no longer works

---

## 6️⃣ Digital Signatures (Deep Concept)

### ✍️ Definition:

A cryptographic method to verify:

* Integrity
* Authentication
* Non-repudiation

---

### 🧠 Analogy:

👉 Like a **signed contract or wax seal**

* Only the sender can sign it
* Anyone can verify it

---

## 7️⃣ Digital Signature Process

### 🧠 Step-by-Step:

### Sender (Alice):

1. Create message (plaintext)

2. Hash the message

3. Encrypt hash with **private key**
   👉 This = digital signature

4. Send:

* Plaintext
* Digital signature

---

### 🧠 Analogy:

👉 Write a letter → stamp it with your **unique seal**

---

### Receiver (Bob):

1. Decrypt signature using **public key**
2. Get original hash
3. Hash received message
4. Compare both hashes

---

### 🧠 Analogy:

👉 Check the seal → compare fingerprints

---

### ✅ If hashes match:

* Message unchanged
* Sender verified

---

### 🧠 Analogy:

👉 Seal is valid → letter wasn’t tampered

---

## 8️⃣ What Digital Signatures Provide

| Feature         | Meaning                    |
| --------------- | -------------------------- |
| Integrity       | Message not altered        |
| Authentication  | Confirms sender identity   |
| Non-repudiation | Sender cannot deny sending |

---

### 🧠 Analogy:

* Integrity → fingerprint matches
* Authentication → verified signature
* Non-repudiation → signed contract you can’t deny

---

## 9️⃣ Key Takeaways

* Hashing = **one-way fingerprint**
* Small change → big difference
* Collisions = bad (avoid weak algorithms)
* Passwords stored as **salted hashes**
* Digital signatures use:

  * Hashing + encryption
* Private key signs, public key verifies

---

### 🧠 Analogy:

👉 System = identity verification system

* Hash = fingerprint
* Signature = legal signature

---

## 🧾 Flashcards (Active Recall)

1. **Q:** What is a hash?
   **A:** A fixed-length fingerprint of data.

2. **Q:** Can a hash be reversed?
   **A:** No.

3. **Q:** What is a collision?
   **A:** Two different inputs producing the same hash.

4. **Q:** Why is MD5 insecure?
   **A:** It has known collision vulnerabilities.

5. **Q:** What is salting?
   **A:** Adding random data before hashing.

6. **Q:** What is a rainbow table?
   **A:** Precomputed hashes used to crack passwords.

7. **Q:** What key is used to sign a digital signature?
   **A:** Private key.

8. **Q:** What key verifies a digital signature?
   **A:** Public key.

---

## ✅ Final Insight

👉 Hashing = integrity
👉 Digital signatures = integrity + identity

If hashes don’t match → **DO NOT TRUST THE DATA**
