# 🔧 1.2 : Technical Control Management

---

## 1️⃣ Big Picture

**Key Idea:**
Change management decides **what should change**
👉 The technician executes **how the change is done**

In large environments:

* Changes affect **hundreds or thousands of systems** 

---

## 2️⃣ Allow List vs Deny List

### ✅ Allow List (Whitelisting)

* Only approved applications can run
* Everything else is **blocked**

👉 **Most secure**

---

### ❌ Deny List (Blacklisting)

* Everything runs EXCEPT blocked apps
* Used by:

  * Antivirus
  * Anti-malware

👉 **More flexible, less secure**

---

### 🧠 Exam Tip:

* **Allow list = strict security**
* **Deny list = broader access**

---

## 3️⃣ Scope Control (VERY IMPORTANT)

When implementing a change:

* You can ONLY do what’s in the **approved scope**

### 🚫 Example:

* Approved: Update printer drivers
* ❌ Not allowed: Update other apps just because you have time

---

### ⚠️ Exception:

* If needed to complete the task
* Example:

  * Updating drivers requires config file change → allowed

👉 Must follow documented policies

---

## 4️⃣ Downtime & Maintenance Windows

### ⏰ Best Practice:

* Perform changes during:

  * Nights
  * Weekends
  * Low usage periods

---

### 🏢 24/7 Environments:

* Use **failover systems**:

  * Switch to backup system
  * Update primary
  * Switch back

👉 Minimizes downtime

---

## 5️⃣ System Restart / Service Restart

After changes, you may need:

### 🔄 Full Reboot

* OS restart
* Device power cycle

### ⚙️ Service Restart

* Restart specific service (faster)

### 👤 User Restart

* Close and reopen app

---

## 6️⃣ Legacy Systems (HIGH RISK)

### 🧓 Definition:

Old systems:

* No longer supported
* Poorly understood

### ⚠️ Risks:

* Breaking system
* No vendor support

### 💡 Solution:

* Document everything
* Understand dependencies

---

## 7️⃣ Dependencies (VERY TESTABLE)

### 🔗 Definition:

One system depends on another

### Examples:

* App needs another service installed first
* Firewall update required before management software

👉 Order matters!

---

## 8️⃣ Documentation (CRITICAL)

Every change must update:

* Network diagrams
* Configurations
* Procedures

👉 Without updates:

* Documentation becomes useless quickly

---

## 9️⃣ Version Control

### 📚 Definition:

Tracks changes over time

### Benefits:

* See what changed
* Revert to previous version
* Maintain history

### Examples:

* Config files
* OS patches
* Application updates

---

## 🔟 Change Flow (Technician View)

1. Receive approved change
2. Follow **strict scope**
3. Implement during maintenance window
4. Apply updates
5. Restart system/service if needed
6. Verify functionality
7. Update documentation
8. Track version changes

---

## 🧠 Memory Anchors

* 📋 **Scope = rules you cannot break**
* ⏰ **Maintenance window = safe time**
* 🔗 **Dependencies = order matters**
* 🔄 **Restart = activate change**
* 📚 **Version control = time machine**

---

## 🧾 Flashcards (Active Recall)

1. **Q:** What is an allow list?
   **A:** Only approved applications can run.

2. **Q:** What is a deny list?
   **A:** Blocks specific applications, allows everything else.

3. **Q:** Why is scope important in change control?
   **A:** Limits changes to approved actions only.

4. **Q:** What is a dependency?
   **A:** When one system or service relies on another.

5. **Q:** Why are maintenance windows used?
   **A:** To reduce impact on users.

6. **Q:** What is version control?
   **A:** Tracking and managing changes to systems/configurations.

7. **Q:** Why are legacy systems risky?
   **A:** They are outdated, unsupported, and hard to modify.

---

## ✅ Final Insight

👉 **Change control = discipline**

Most failures happen when:

* Scope is ignored
* Dependencies are missed
* No rollback/version tracking
